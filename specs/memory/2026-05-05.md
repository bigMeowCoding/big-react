# 2026-05-05

## 发现

- 项目内 TypeScript 文件仅分布在 `react`、`react-dom` 与 `shared` 包的入口与基础实现中。
- JS 迁移后，ESM 导入路径需显式包含 `.js` 扩展名，否则 ESLint 在解析 `packages/react/index.js` 时会报模块未找到。

## 决策

- 项目统一改为 JavaScript：删除 `.ts` 文件与 `tsconfig.json`，同步去除 TypeScript 相关依赖与 ESLint TS 配置。
- 保持现有包结构不变，只做最小改造以确保 demos 和 lint 可继续运行。

## 项目现状沉淀

- 包依赖链路已跑通：`demos -> react-dom -> react-reconciler -> shared`，`react` 提供 JSX 工厂与元素结构，`react-dom` 通过 `createRoot().render()` 驱动渲染。
- `react-reconciler` 当前完成了最小可用渲染闭环：`updateContainer` 入队更新、`workLoop` 构建 workInProgress 树、`completeWork` 生成宿主节点、`commitWork` 执行 `Placement` 挂载。
- 当前 Fiber/调度模型仍是“同步 + 单更新”实现：更新队列仅保存一个 `pending`，调度直接同步执行，适合作为教学阶段的最小基线。
- 已实现节点类型以 `HostRoot/HostComponent/HostText` 为主，`FunctionComponent` 仅完成 tag 与 Fiber 建模，尚未进入函数组件执行与 hooks 流程。
- 副作用系统已具备基础形态（`flags/subtreeFlags/MutationMask`），但 `Update`、`ChildDeletion` 等分支还未在 commit 阶段完整落地。
- 代码中仍有较多调试日志（如 `jsx`、`workLoop`、`completeWork`），后续可以按里程碑逐步收敛，避免干扰 demos 验证输出。

## 下一步建议

- 优先补齐更新路径：在协调阶段复用旧 Fiber，按 key/type 处理可更新场景，并打通 `Update` flag 的 commit 逻辑。
- 完成删除路径：在 diff 阶段收集待删子树并在 commit 阶段统一执行 `ChildDeletion`。
- 引入函数组件执行通路：在 `beginWork` 中支持 `FunctionComponent`，建立最小 hooks（如 `useState`）所需的数据结构与调度回路。
